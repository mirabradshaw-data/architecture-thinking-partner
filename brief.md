# Brief — the problem this solves

**What's broken.** I run two working and load-bearing ICM systems (personal and work) with more I want to build. I've already developed a set of cross-system architectural decisions that I want to maintain, while keeping each build internally consistent and coherent. Executing that well is challenging. By default, the model uses first principles, so building effectively means keen architectural oversight and handholding throughout. And on a bad day, drift slips in.

**What I've tried.** Bringing that discipline myself, build by build — carrying the cross-system decisions and each system's conventions and guiding the model firmly. It works when I'm sharp. It's fragile when my mind is elsewhere and I forget decisions that I've made before, and the cost lands downstream as a clean-up job.  Initially, it was a technique — it supported my building in ICM and developing architecture for it.  Now it's slowing me down. I'm more confident in my taste; it's time for an alternative.

**What I need.** A thinking partner for architecture that forces the discipline: designs within each system's conventions and references my cross-cutting decisions rather than working from scratch, and creates a clear record and a build plan. A model is well-positioned to think through the detail if handed the right context regarding my existing structure, principles and architectural taste. It's a critical piece of the puzzle — it sets up a foundation for the next stage of this workflow — the downstream model-vs-model loops that iterate and improve a build prior to my review gate. It pairs with the PRD interviewer that comes before it, so I'm shipping them together.
